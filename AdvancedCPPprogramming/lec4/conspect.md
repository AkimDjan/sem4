### Двусвязный список
Его длина вычислится за один проход (пока не наткнемся на **nullptr**)

Петля в списке - можем выявить через два прохода, один через 1 нод ходит, другой через соседние, не встретились - тогда петля образовалась (Алгоритм Флойда, 2 указателя)

Что привело к петле? У нас был прямой доступ к полям списка, и мы могли конец с началом сопоставить

Поэтому нам необходимо инкапсулировать поля (**private**), 

```cpp
template <typename T> struct list_t{
private:
    struct node_t;
    node_t *top_, *back_;
public:
    size_t length() const;
}
```
Также есть механизм friend-класса, может обойти private - у дружественного класса есть доступ к приватным полям

Это не значит, что надо повсеместно этим пользователем. Язык дает эту возможность например для того, чтобы например тестировать 

Постулат: в хорошо спроектированной программе нет "друзей" (кроме тестов)

В Классе по умолчанию все поля приватные, в том числе и наследование у нас по умолчанию private
```cpp
template <typename T> class list_t{
    struct node_t;
    node_t *top_, *back_;
public:
    size_t length() const;
}
```

### Ссылки
Что можно сделать с указателем и нельзя делать с ссылкой? Указатель мы можем поменять, а ссылку не можем :)

Типы читаем справа налево

```cpp
int *q =*reinterpret_cast<int **>(&id); // указатель на указатель на инт
int * const p; // константный указатель на инт
float* (*()) q; //сначала читаем от внутренней звездочки - указатель на функцию и возвращающую флоат
float* *() q1; // функция которая возвращает указатель на указатель на флоат
```
Инкапсуляция - свойстов типов, а не объектов

Инкапсуляция критически требует конструкторов и деструкторов

Разные шаблонные параметры - разные типы
